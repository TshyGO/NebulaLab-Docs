NebulaLab 贡献结算规则

本规则用于规范 NebulaLab 贡献结算、公共支出预算及结余处理。结算比例以 GitHub 已合并的有效改动为主要依据，并由统一脚本按固定口径计算。

1. 目的 2. 资金池 3. 跨年度结算 4. 工作量积分 5. 排除项 6. Squash 与多人协作 7. 数据维护 8. 执行口径

1. 目的

本规则用于明确 NebulaLab 贡献结算、公共支出预算和结余处理方式，确保项目公共支出具备稳定预算， 贡献者分配具备明确依据，并降低人工评审成本。

NebulaLab 的开发协作主要发生在 GitHub。考虑到团队成员普遍使用 AI 辅助开发，逐项评定 Issue 或 PR 难度缺乏稳定标准，且会增加维护负担。因此，贡献结算以 GitHub 已合并的有效改动为主要依据。

本规则不以 commit 数或原始增删行作为单一依据。commit 数主要反映提交频率，原始增删行容易受到 lock 文件、自动生成文件、格式化改动及机械性重构影响。正式结算采用有效工作量积分。

2. 资金池

每次年度赞助到账后，先扣除必要运维开支。

可分配余额 = 赞助到账金额 - 必要运维开支
年度结算池 = 可分配余额 × 50%
项目公共池 = 可分配余额 × 50%

年度结算池用于按贡献比例向贡献者分配。项目公共池用于项目公共支出，包括模型调用费用、服务器、域名、工具订阅、CI / 构建资源、临时协作支出及其他必要运营支出。

3. 跨年度结算

从第二个赞助周期开始，每次新赞助到账时，将上一周期公共池结余并入本年度结算池。

本年度结算池 = 上一周期公共池结余 + 本年度新赞助扣除必要运维后的 50%

本年度结算池按上一赞助周期内的有效工作量积分分配。例如，2027 年 5 月获得新赞助时，2027 年度结算池按 2026-05 到 2027-05 的 GitHub 有效工作量积分分配。

4. 有效工作量积分

只统计已合并进入主分支的非 merge commit。计算步骤：

  1. 用 git log --no-merges -M -C --numstat 读取每个纳入结算仓库主分支 commit；-M -C 让 rename / copy 不被识别为先删后增。
  2. 解析每个 commit message 中的 Co-authored-by: 行，commit 的有效改动量在 primary author 与所有 co-authors 之间均分。
  3. 通过 identity_aliases 归并同一贡献者的不同 Git 身份；通过 bot_names 与 [bot] 启发式排除自动化提交与 AI 助手 trailer。
  4. 按文件类别为增删行设置权重；新增行按 1 倍计入，删除行按 0.5 倍计入。
  5. 按 (作者, ISO 周, 仓库) 三维聚合 weighted churn，对聚合值开方后乘以仓库权重得到该桶得分。
  6. 汇总每位贡献者所有桶的得分，按占比分配。
文件类型权重
核心代码 (src/、engine/、modules/、app/、web/、src-tauri/src/、.css/.html/.vue)1.0
测试代码 (tests/、*.test.*、*.spec.*)0.8
配置 / CI / 脚本 (.github/、scripts/、supabase/、*.config.ts)0.6
文档 (docs/、README*、*.md/.mdx/.rst)0.4
未分类 (fallback)0.0
排除项 (lock / 构建产物 / vendored)0.0
仓库权重
NebulaLab 主产品仓库1.0
NebulaLab Plugins 插件库0.9
NebulaLab Docs 用户手册 / 文档仓库0.45
 
file_weighted_churn = (additions + 0.5 × deletions) × file_type_weight
bucket_churn(author, week, repo) = Σ (1 / N_authors) × file_weighted_churn
bucket_score(author, week, repo) = sqrt(bucket_churn) × repo_weight
个人周期积分 = Σ over (week, repo) of bucket_score(self, week, repo)
个人分配比例 = 个人周期积分 / 全部贡献者周期积分之和

为什么按周聚合：如果按 commit 直接开方，1000 行工作拆成 10 个 100 行 commit 可以得到 10 × √100 = 100 分，而一次性提交只有 √1000 ≈ 31.6 分，算法会变相鼓励 commit 切片。按 (作者, ISO 周, 仓库) 聚合后，同一作者在同一周内的工作先合并、再开方，拆分对最终结果几乎没有影响。

5. 默认排除项

以下内容默认 weight = 0，不计入有效工作量：

  • package-lock.json、Cargo.lock、yarn.lock、pnpm-lock.yaml、poetry.lock、uv.lock、Pipfile.lock、composer.lock
  • 后缀 .min.js、.min.css、.map、.lock 的文件
  • 路径前缀 dist/、build/、out/、vendor/、third_party/、node_modules/、src-tauri/gen/、assets/data/

以下两类无法自动识别，需要由维护者在 PR review 阶段人工标记，必要时在合入前手动剔除：

  • 纯格式化 / 自动重排版改动
  • 一次性引入的自动生成 schema 或迁移产物

如果某次被排除文件的修改确实代表重要工程工作，例如依赖体系迁移或构建系统升级，可在结算时单独说明并人工确认是否部分计入。

6. Squash merge 与多人协作

NebulaLab 默认使用 GitHub squash merge。squash 之后，PR 内的多个 commit 会被合并为 main 上的一个 commit，结果 commit 的 author 字段只能保留一人。

为避免多人协作的非主作者贡献丢失，多人协作的 PR 必须在 commit message 中保留 Co-authored-by: trailer： 

Co-authored-by: Name <name@example.com>

结算脚本会解析这些 trailer，并将该 commit 的 weighted churn 在 primary author 与所有 co-authors 之间均分。GitHub 在 squash merge 时会自动把 PR 内各 commit 的非 primary author 提为 Co-authored-by:，但如果开发者使用 git rebase / 本地 squash 时手动改写过 commit message，请确认 trailer 仍然存在。

AI 助手（Claude、Codex 等）若被自动写入 Co-authored-by:，会被 contribution-identity.json 中的 bot_names 过滤，不参与积分分配。

Nebula Graph Update Bot <bot@nebula.graph> 是例外：它是 TshyGO 控制的 AI-agent 直接提交兜底身份，因此在 identity_aliases 中归并到 TshyGO。当 commit 能通过 GitHub API 解析到已合并 PR 作者时，pr_author_login 会优先覆盖这个 git author 身份，确保其他贡献者通过共享 bot identity 合入的 PR 仍归属到实际 PR 作者。

7. 数据公开与维护

贡献统计由 scripts/contribution_metrics.py 自动生成，并输出结构化统计数据。统计数据包含必要的 commit 明细，因此网页可以按全部历史、年度、月份、自定义日期段或仓库重新计算贡献比例。

身份归并 (identity_aliases) 与 Bot 列表 (bot_names) 集中在 docs/sponsorship-policy/contribution-identity.json，新增贡献者或 bot 不需要改 Python 代码。仓库列表与权重集中在 docs/sponsorship-policy/contribution-repos.json。

统计数据不作秒级实时刷新。页面筛选在浏览器中即时重算；底层 metrics.json 由 sponsorship-policy workflow 生成后推送到 TshyGO/NebulaLab-Docs 的 docs/public/sponsorship-policy/，不会回写到主应用仓库。该发布步骤依赖 RELEASE_REPO_TOKEN，token 应仅授予目标 companion repo 所需的 contents: write 权限；主仓 workflow 自身只需要 contents: read 与 pull-requests: read。docs/sponsorship-policy/metrics.json 被 .gitignore 忽略，本地预览需手动运行脚本生成。

规则文档和统计页面由仓库维护者管理。其他成员如需提出修改建议，应通过 Issue、Discussion 或其他明确记录渠道提出。

8. 执行口径

每次年度赞助到账并扣除必要运维开支后，其中 50% 与上一赞助周期公共池结余合并为本年度结算池，按上一赞助周期内 GitHub 已合并有效工作量积分分配；剩余 50% 进入新的项目公共池，用于下一赞助周期内的公共支出。

有效工作量积分基于已合并 commit 自动计算。统计时排除 Bot、lock 文件、自动生成文件、构建产物与 vendored 代码，并按代码、测试、文档、配置等类型设置文件权重，同时按主仓库、插件库和 Docs 仓库设置仓库权重。删除行按新增行的 50% 计入；rename / copy 通过 git -M -C 单算；多人协作通过 Co-authored-by: 在贡献者间均分。最终按 (作者, ISO 周, 仓库) 聚合后开方，再乘以仓库权重，得到个人周期积分。

本页面为 NebulaLab 贡献结算规则的网页版本，由仓库维护者统一维护。
