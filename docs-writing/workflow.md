# AI-Assisted Manual Workflow

This workflow is for multi-person documentation work on Nebula Lab.

## Principle

AI may produce a full page and open a PR, but it must not invent facts. Each generated page must be grounded in a fact pack and reviewed by a human before merge.

The desired flow is:

```text
human picks page
AI extracts facts
AI writes page brief
cheap model drafts page
cheap model rewrites tone
strong reviewer checks facts
build runs
AI opens PR
human reviews PR
```

## Inputs

Each page should use these inputs:

- `docs-writing/style-guide.md`
- `docs-writing/terminology.md`
- the page's fact pack in `docs-writing/fact-packs/`
- the page brief in `docs-writing/page-briefs/`
- existing docs under `docs/`

When the app source repository is available, fact extraction may also use:

- `src/components/`
- `src/lib/copy.ts`
- `src/api/`
- `src/stores/`
- `release-notes.md`
- `docs/PUBLIC_RELEASE_REPO_README.md`

## Stage 1: Fact Pack

The fact pack is the source of truth for one page. It should answer:

- where the feature appears in the app,
- what the user can do,
- which UI labels are visible,
- what the prerequisites are,
- what must not be claimed,
- which source files or docs support the facts.

Fact packs may be generated by AI, but they must cite concrete files or visible UI text.

## Stage 2: Page Brief

The page brief tells the drafting model what to write. It should include:

- target reader,
- user problem,
- page outline,
- required facts,
- exclusions,
- links to related pages.

The brief should be short enough for a cheaper model to follow without needing repo access.

## Stage 3: Draft

The draft model writes one page only. It receives:

- style guide,
- terminology,
- page brief,
- fact pack,
- page template.

If information is missing, it should write `需要确认：...` rather than making up behavior.

## Stage 4: Human-Tone Rewrite

The rewrite model removes generic AI prose and makes the page usable:

- shorter sentences,
- concrete steps,
- fewer abstract nouns,
- no marketing language,
- no vague "according to your needs" filler.

The rewrite must not add new product facts.

## Stage 5: Fact Check

Before opening a PR, check:

- every button or page name exists in the fact pack or app copy,
- no future roadmap item is described as shipped,
- Pro-only or external-tool features state prerequisites,
- links are valid,
- `npm run docs:build` passes.

## Stage 6: Pull Request

Use one PR per page or small page group. The PR body must include:

- source fact pack,
- generated or edited pages,
- validation command,
- human review focus.

Do not merge generated documentation without human review unless the repository owner explicitly requests an admin bypass.
